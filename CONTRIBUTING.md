# Contributing Guidelines

Here are a list of guidelines, from the most important (游댮), to the least important (游댯)

>   游댮 - very important  
>   游 - important  
>   游리 - nice to have  
>   游릭 - you dont have to do that, your'e always welcome to  
>   游댯 - optional, but greatly appreciated :D  


游댮 No single-letter variables/functions  
游댮 Don't change any API without specifying the change  
游댮 Don't create new top-level subfolders. If subfolders are needed, place them in `vision.ds` or `vision.helpers`  
游 Don't use abbreviations  
游 Don't write module-level functions/fields exclusively. Try using classes whenever possible
游 Don't remove any feature if you don't give an alternative  
游 When heavily editing core API's (`Vision.hx`, `Image.hx`), make sure to specify your reason for editing  
游 Vision has a specific formatting style, specified in `hxformat.json`. Don't write code that doesn't conform to the formatting rules. It makes things look messy.  
游리 When creating a fork, prefix your branch's name with `feature/` for a new feature, `qol/` for a QOL change/addition, `perf/` for a performance improvement, and `impr/` for an improvement upon an existing feature   
游리 Write self-descriptive code (full variable names, does'nt matter how long they are)  
游리 Don't write obfuscated code (even if its faster, if it isn't understandable, its hard to maintain)  
游릭 Write comments to explain your code  
游댯 Add documentation  
游댯 Extend/fix documentation
